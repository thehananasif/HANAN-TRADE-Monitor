Ships 图层通过 AIS（Automatic Identification System，自动识别系统）数据提供实时船舶追踪和海域态势感知，监控关键咽喉要道、检测异常船舶行为，并通过 WebSocket 连接流式传输位置更新。

## Route Explorer

Route Explorer 是一个键盘优先的工作流，用于规划任意两国之间的航运，并查看咽喉要道暴露情况、绕行走廊、陆上替代方案以及各国影响——所有这些都叠加在海事情报图层之上。完整的工作流、标签页内容、键盘绑定和分层门控请参见专门的 [Route Explorer](/zh/route-explorer) 页面。

## 咽喉要道监控

系统监控 13 条战略水道，这些水道的中断可能影响全球贸易，由三个数据源驱动：IMF PortWatch（每周船舶过境计数）、AISStream（实时 24 小时穿越计数器）和 CorridorRisk（风险情报）。

| 咽喉要道 | 战略重要性 |
|------------|---------------------|
| **霍尔木兹海峡** | 全球 20% 石油过境；伊朗控制 |
| **苏伊士运河** | 欧亚航运；单点故障 |
| **马六甲海峡** | 亚太地区主要石油航线 |
| **曼德海峡** | 红海通道；也门/胡塞活动 |
| **巴拿马运河** | 美洲东西向过境 |
| **台湾海峡** | 半导体供应链；解放军活动 |
| **好望角** | VLCC 的苏伊士绕行航线 |
| **直布罗陀海峡** | 大西洋-地中海门户；北约咽喉要道 |
| **博斯普鲁斯海峡** | 黑海通道；包含达达尼尔海峡走廊；《蒙特勒公约》 |
| **朝鲜海峡** | 日韩贸易；东亚最繁忙走廊 |
| **多佛尔海峡** | 世界最繁忙航运通道 |
| **刻赤海峡** | 俄罗斯控制；乌克兰经亚速海的粮食运输受限 |
| **龙目海峡** | 大型油轮的马六甲绕行通道 |

每个咽喉要道卡片显示实时过境计数（油轮 vs 货轮）、环比变化，以及使用 TradingView lightweight-charts 渲染的可展开 180 天时间序列图表。

## 密度分析

船舶位置被聚合到 2 度网格中以计算交通密度。每个网格单元跟踪：

- 当前船舶数量
- 历史基线（30 分钟滚动窗口）
- 相对基线的变化百分比

密度变化 +/-30% 会触发警报，表明可能存在拥堵、改道或封锁。

## 暗船检测

系统监控 AIS 间隙，即停止发送位置的船舶。在监控区域中，超过 60 分钟的 AIS 间隙可能表明：

- 规避制裁（船对船转运）
- 非法捕捞
- 军事活动
- 设备故障

间隙后重新出现的船舶会在整个会话期间被标记。

## WebSocket 架构

AIS 数据通过 WebSocket relay 流式传输，实现无需轮询的实时更新：

```
AISStream -> WebSocket Relay -> Browser
              (ws://relay)
```

连接断开时会自动重连，退避间隔为 30 秒。当 Ships 图层被禁用时，WebSocket 会断开连接以节省资源。

## Railway Relay 架构

某些 API 会阻止来自云服务商（Vercel、AWS、Cloudflare Workers）的请求。Railway relay 服务器提供经过认证的访问：

```
Browser -> Railway Relay -> External APIs
            (Node.js)      (AIS, OpenSky, RSS)
```

**Relay 功能**：

| 端点 | 用途 | 认证方式 |
|----------|---------|----------------|
| `/`（WebSocket） | AIS 船舶流 | AISStream API key |
| `/opensky` | 军用飞机 | OAuth2 Bearer token |
| `/rss` | 被阻止的 RSS feeds | 无（user-agent 伪装） |
| `/health` | 状态检查 | 无 |

**环境变量**（Railway）：

- `AISSTREAM_API_KEY` - AIS 数据访问
- `OPENSKY_CLIENT_ID` - OAuth2 客户端 ID
- `OPENSKY_CLIENT_SECRET` - OAuth2 客户端密钥

**为什么选择 Railway？**

- 住宅 IP 段（不像云服务商那样被阻止）
- 支持持久连接的 WebSocket
- 全球边缘部署，低延迟
- 免费层足以应对中等流量

relay 是无状态的；它仅进行认证和代理请求。所有缓存和处理都在客户端或 Vercel Edge Functions 中进行。

破坏评分方法论另请参见 [金融数据 - 咽喉要道](/zh/finance-data)。
